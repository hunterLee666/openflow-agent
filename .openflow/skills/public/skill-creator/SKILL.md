# 技能创建者

用于创建新技能和迭代改进它们的技能。

从高层次来看，创建技能的过程如下：

- 决定你想要技能做什么以及大致如何做
- 撰写技能的初稿
- 创建几个测试提示并在它们上运行带技能的 claude
- 帮助用户定性和定量地评估结果
  - 当运行在后台进行时，如果没有评估，创建一些定量评估（如果有一些，你可以直接使用或修改如果你觉得需要改变什么）。然后向用户解释它们（或者如果它们已经存在，解释已经存在的）
  - 使用 `eval-viewer/generate_review.py` 脚本向用户显示结果供他们查看，也让它们查看定量指标
- 根据用户对结果的评估反馈重写技能（也从定量基准中出现的任何明显缺陷）
- 重复直到你满意
- 扩大测试集并再次尝试更大规模

使用此技能时，你的工作是弄清楚用户在这个过程的哪个阶段，然后介入并帮助他们完成这些阶段。例如，也许他们像是"我想为一个 X 创建一个技能"。你可以帮助缩小他们所说的范围、撰写草稿、编写测试用例、弄清楚他们想要如何评估、运行所有提示，并重复。

另一方面，也许他们已经有了技能的草稿。在这种情况下，你可以直接进入评估/迭代循环。

当然，你应该始终灵活，如果用户像是"我不需要运行一堆评估，只是和我一起感受"，你可以这样做。

然后在技能完成后（但同样，顺序是灵活的），你也可以运行技能描述改进器，我们有单独的脚本，用于优化技能的触发。

明白了吗？明白了。

## 与用户沟通

技能创建者可能被对编码术语熟悉程度差异很大的用户使用。如果你不熟悉（你怎么知道，这是最近才开始的趋势），现在有一种趋势，Claude 的力量激励着管道工打开他们的终端、父母和祖父母谷歌"如何安装 npm"。另一方面，大部分用户可能相当精通计算机。

所以请注意上下文线索来理解如何措辞你的沟通！默认情况下，给你一些概念：

- "evaluation"和"benchmark"是边界，但可以
- 对于"JSON"和"assertion"，你需要看到用户确实知道这些东西是什么的认真线索，然后才不加解释地使用它们

如果有疑问，可以简要解释术语，如果不确定用户是否能理解，可以自由地用简短定义澄清术语。

## 创建一个技能

### 捕获意图

首先理解用户的意图。当前的对话可能已经包含用户想要捕获的工作流程（例如，他们说"把这个变成一个技能"）。如果是，从对话历史中首先提取答案——使用的工具、步骤顺序、用户所做的更正、观察到的输入/输出格式。用户可能需要填补空白，并在继续下一步之前确认。

1. 这个技能应该让 Claude 做什么？
2. 这个技能什么时候应该触发？（什么用户措辞/上下文）
3. 期望的输出格式是什么？
4. 我们应该设置测试用例来验证技能是否有效吗？具有客观可验证输出的技能（文件转换、数据提取、代码生成、固定工作流程步骤）受益于测试用例。主观看输出的技能（写作风格、艺术）通常不需要。为技能类型建议适当的默认值，但让用户决定。

### 面试与研究

主动询问边缘情况、输入/输出格式、示例文件、成功标准和依赖关系。等到这部分敲定后再编写测试提示。

检查可用的 MCP——如果对研究有用（搜索文档、查找类似技能、查找最佳实践），如果可用，通过子代理并行研究，否则内联研究。带着上下文准备，以减少用户的负担。

### 撰写 SKILL.md

基于用户面试，填写这些组件：

- **name**：技能标识符
- **description**：何时触发，做什么。这是主要的触发机制——包括技能做什么和具体触发上下文。所有"何时使用"信息都在这里，不在正文。注意：目前 Claude 有"触发不足"的倾向——在技能有用时不使用它们。为了解决这个问题，请把技能描述写得更"有推动性"一点。例如，不要写"如何构建一个简单的快速仪表板来显示内部 Anthropic 数据"，你可以写"如何构建一个简单的快速仪表板来显示内部数据。每当用户提到仪表板、数据可视化、内部指标或想要显示任何类型的公司数据时，确保使用此技能，即使他们没有明确要求'仪表板'。"
- **compatibility**：必需工具、依赖项（可选，很少需要）
- **技能的其余部分 :)**

### 技能撰写指南

#### 技能结构

```
skill-name/
├── SKILL.md（必需）
│   ├── YAML frontmatter（name, description 必需）
│   └── Markdown 指令
└── 捆绑资源（可选）
    ├── scripts/    - 用于确定性/重复性任务的可执行代码
    ├── references/ - 根据需要加载到上下文的文档
    └── assets/     - 输出中使用的文件（模板、图标、字体）
```

#### 渐进式披露

技能使用三级加载系统：
1. **元数据**（name + description）——始终在上下文中（~100字）
2. **SKILL.md 正文**——每当技能触发时在上下文中（理想情况下 <500 行）
3. **捆绑资源**——按需（无限，脚本可以执行而不加载）

这些字数是近似的，如果有需要，你可以随意更长。

**关键模式：**
- 保持 SKILL.md 在 500 行以下；如果你接近这个限制，添加额外的层次结构，并提供清晰的指针说明使用技能的人应该去哪里下一步。
- 从 SKILL.md 清楚地引用文件，并提供关于何时阅读它们的指导
- 对于大型参考文件（>300 行），包含目录

**领域组织**：当技能支持多个领域/框架时，按变体组织：
```
cloud-deploy/
├── SKILL.md（工作流程 + 选择）
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claude 只阅读相关参考文件。

#### 缺乏惊喜原则

这是不言而喻的，但技能不能包含恶意软件、利用代码或任何可能损害系统安全的内容。技能的内容不应该在用户意图上让他们感到意外。不要附和创建误导性技能或旨在促进未经授权访问、数据泄露或其他恶意活动的技能的请求。但是像"扮演 XYZ"这样的角色扮演是可以的。

#### 撰写模式

在指令中优先使用祈使形式。

**定义输出格式**——你可以这样写：
```markdown
## 报告结构
始终使用此确切模板：
# [标题]
## 执行摘要
## 关键发现
## 建议
```

**示例模式**——包含示例很有用。你可以这样格式化（但如果示例中有"输入"和"输出"，你可能想要稍微不同）：
```markdown
## 提交消息格式
**示例1：**
输入：使用 JWT 令牌添加用户身份验证
输出：feat(auth): implement JWT-based authentication
```

### 撰写风格

试着向模型解释为什么事情是重要的，而不是用沉重的 MUST。与其使用超级狭窄的特定示例，不如尝试使用心智理论并使技能通用化。首先写一份草稿，然后用新鲜的眼光看它并改进它。

### 测试用例

撰写技能初稿后，想出 2-3 个现实的测试提示——真正的用户实际会说的那种。与用户分享它们：[你不必使用完全相同的语言] "这里有几个我想尝试的测试用例。这些看起来对吗，或者你想添加更多吗？"然后运行它们。

将测试用例保存到 `evals/evals.json`。还不要写断言——只有提示。你会在下一步（当运行在进行中时）起草断言。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "User's task prompt",
      "expected_output": "Description of expected result",
      "files": []
    }
  ]
}
```

参见 `references/schemas.md` 获取完整模式（包括你稍后添加的 `assertions` 字段）。

## 运行和评估测试用例

本节是一个连续的序列——不要半途而废。不要使用 `/skill-test` 或任何其他测试技能。

将结果放入 `<skill-name>-workspace/` 作为技能目录的同级。在工作空间内，按迭代（`iteration-1/`、`iteration-2/` 等）组织，在其中每个测试用例得到一个目录（`eval-0/`、`eval-1/` 等）。不要预先创建所有这些——按需创建目录。

### 步骤1：在同一轮中生成所有运行（有技能 AND 基线）

对于每个测试用例，在同一轮中生成两个子代理——一个有技能，一个没有。这很重要：不要先生成有技能的运行，然后再回来生成基线。立即启动所有内容，这样它们都会在大致相同的时间完成。

**有技能运行：**

```
执行此任务：
- 技能路径：<path-to-skill>
- 任务：<eval prompt>
- 输入文件：<eval files if any, or "none">
- 保存输出到：<workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 保存的输出：<what the user cares about — e.g., "the .docx file", "the final CSV">
```

**基线运行**（相同提示，但基线取决于上下文）：
- **创建新技能**：根本没有技能。相同提示，无技能路径，保存到 `without_skill/outputs/`。
- **改进现有技能**：旧版本。在编辑之前，快照技能（`cp -r <skill-path> <workspace>/skill-snapshot/`），然后将基线子代理指向快照。保存到 `old_skill/outputs/`。

为每个测试用例编写一个 `eval_metadata.json`（断言现在可以为空）。给每个评估一个基于它测试什么的描述性名称——不仅仅是"eval-0"。也用它作为目录名。如果此迭代使用新的或修改的评估提示，为每个新评估目录创建这些文件——不要假设它们从前一个迭代继承。

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "The user's task prompt",
  "assertions": []
}
```

### 步骤2：在运行进行时，起草断言

不要只是等待运行完成——你可以有效地利用这段时间。为每个测试用例起草定量断言，并向用户解释它们。如果 `evals/evals.json` 中已经存在断言，审查它们并解释它们检查什么。

好的断言是客观可验证的，并且有描述性名称——它们应该在基准查看器中清晰地阅读，这样有人一眼就能理解每个检查什么。主观看技能（写作风格、设计质量）最好定性评估——不要把断言强加于需要人类判断的事情上。

一旦起草了就更新 `eval_metadata.json` 文件和 `evals/evals.json` 中的断言。还要向用户解释他们在查看器中会看到什么——定性和定量基准。

### 步骤3：当运行完成时，捕获计时数据

当每个子代理任务完成时，你收到包含 `total_tokens` 和 `duration_ms` 的通知。立即将数据保存到运行目录中的 `timing.json`：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这是捕获此数据的唯一机会——它通过任务通知传来，不会被其他地方保留。当每个通知到达时处理它，而不是试图批量处理。

### 步骤4：评分、聚合并启动查看器

所有运行完成后：

1. **对每个运行评分**——生成一个评分子代理（或内联评分），读取 `agents/grader.md` 并根据输出评估每个断言。将结果保存到每个运行目录中的 `grading.json`。grading.json 期望数组必须使用 `text`、`passed` 和 `evidence` 字段（不是 `name`/`met`/`details` 或其他变体）——查看器依赖这些确切的字段名称。对于可以编程检查的断言，编写并运行脚本而不是目测——脚本更快、更可靠，可以跨迭代重用。

2. **聚合成基准**——从技能创建者目录运行聚合脚本：
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   这会产生 `benchmark.json` 和 `benchmark.md`，其中包含每个配置通过率、时间和代币，以及 mean ± stddev 和 delta。如果手动生成 benchmark.json，请参见 `references/schemas.md` 获取查看器期望的确切模式。
将每个 with_skill 版本放在其基线对应版本之前。

3. **做分析师通过**——读取基准数据并呈现聚合统计可能隐藏的模式。参见 `agents/analyzer.md`（"分析基准结果"部分）了解要查找的内容——比如无论技能如何总是通过的断言（非区分性）、高方差评估（可能不稳定）和时间/代币权衡。

4. **启动查看器**，同时提供定性输出和定量数据：
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   对于迭代 2+，也传递 `--previous-workspace <workspace>/iteration-<N-1>`。

   **同事/无头环境：** 如果 `webbrowser.open()` 不可用或环境没有显示器，使用 `--static <output_path>` 编写独立的 HTML 文件而不是启动服务器。当用户点击"提交所有评论"时，反馈将下载为 `feedback.json` 文件。下载后，将 `feedback.json` 复制到工作空间目录，以便下一次迭代拾取。

注意：请使用 generate_review.py 创建查看器；无需编写自定义 HTML。

5. **告诉用户** 类似："我在你的浏览器中打开了结果。有两个选项卡——'输出'让你点击每个测试用例并留下反馈，'基准'显示定量比较。完成后，回来告诉我。"

### 用户在查看器中看到的内容

"输出"选项卡一次显示一个测试用例：
- **提示**：给出的任务
- **输出**：技能生成的文件，尽可能内联呈现
- **上一个输出**（迭代 2+）：显示上一次迭代输出的折叠部分
- **正式评分**（如果运行了评分）：显示断言通过/失败的折叠部分
- **反馈**：一个随着输入自动保存的文本框
- **上一个反馈**（迭代 2+）：他们在上一次迭代中的评论，显示在文本框下方

"基准"选项卡显示统计摘要：每个配置的通过率、计时和代币使用情况，以及按评估的细分和分析师观察。

导航通过 prev/next 按钮或箭头键。完成后，他们点击"提交所有评论"，将所有反馈保存到 `feedback.json`。

### 步骤5：阅读反馈

当用户告诉你完成后，读取 `feedback.json`：

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "the chart is missing axis labels", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "perfect, love this", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空反馈意味着用户认为没问题。将你的改进集中在用户有具体投诉的测试用例上。

完成后终止查看器服务器：

```bash
kill $VIEWER_PID 2>/dev/null
```

## 改进技能

这是循环的核心。你已经运行了测试用例，用户已经审查了结果，现在你需要根据他们的反馈让技能变得更好。

### 如何思考改进

1. **从反馈中归纳。** 这里发生的大图景是我们试图创建可以使用数百万次（也许是字面上的，甚至更多）的技能，跨许多不同的提示。你和用户在这里一遍又一遍地迭代只有几个例子，因为这有助于加快速度。用户非常了解这些例子，可以快速评估新输出。但是如果你和用户共同开发的技能只适用于那些例子，那它就没用了。不要进行繁琐的过度拟合更改或压迫性的限制性 MUSTS，如果有一些顽固的问题，你可以尝试分支并使用不同的隐喻，或推荐不同的工作模式。尝试相对便宜，也许你会碰到很棒的东西。

2. **保持提示精简。** 删除没有发挥作用的东西。确保阅读了转录本，而不仅仅是最终输出——如果看起来技能让模型浪费了大量时间做无生产力的事物，你可以尝试删除让它那样做的技能部分，看看会发生什么。

3. **解释原因。** 努力解释你要求做的一切的**原因**。今天的 LLM 很聪明。他们有很好的心智理论，当给予良好的框架时可以超越死板指令，真正发挥作用。即使用户的反馈是简短的或沮丧的，也要努力真正理解任务和用户为什么写他们写的，以及他们实际写了什么，然后把这种理解转化为指令。如果你发现自己用所有大写的 ALWAYS 或 NEVER，或者使用超级 rigid 结构，那是黄旗——如果可能，重新框架并解释推理，这样模型理解为什么你要求的事情是重要的。这是一种更人性化、更有力量、更有效的方法。

4. **寻找跨测试用例的重复工作。** 从测试运行中阅读转录本，注意子代理是否都独立编写了类似的辅助脚本或对某事物采取了相同的多步骤方法。如果所有3个测试用例都导致子代理编写了 `create_docx.py` 或 `build_chart.py`，这是技能应该捆绑该脚本的强烈信号。写一次，放在 `scripts/` 中，然后告诉技能使用它。这节省了每次未来调用重新发明轮子。

这个任务相当重要（我们正在这里尝试每年创造数十亿美元的经济价值！）你的思考时间不是瓶颈；慢慢来，真正思考。也许我会建议写一份修订草案，然后用新的眼光看它并改进。真的尽你所能进入用户的头脑，理解他们想要和需要什么。

### 迭代循环

改进技能后：

1. 将你的改进应用到技能
2. 在新的 `iteration-<N+1>/` 目录中重新运行所有测试用例，包括基线运行。如果你是创建新技能，基线始终是 `without_skill`（无技能）——这在迭代中保持不变。如果你是改进现有技能，使用你的判断什么作为基线有意义：用户带来的原始版本，或者上一个迭代。
3. 重复循环
