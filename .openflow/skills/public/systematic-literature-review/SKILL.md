# 系统文献综述技能

## 概述

此技能针对某个研究主题的多篇学术论文生成结构化的**系统文献综述（SLR）**。给定一个主题查询，它搜索 arXiv，从每篇论文中并行提取结构化元数据（研究问题、方法论、关键发现、局限性），综合整个集合的主题，并在最终报告中发出一致的引用。

**与 `academic-paper-review` 的区别：** 该技能对单篇论文进行深度同行评审。本技能对多篇论文进行广度优先的综合。如果用户提供了一篇论文 URL 并要求"审查这篇论文"，请路由到 `academic-paper-review`。

## 何时使用此技能

当用户想要以下内容时使用此技能：

- 关于某个主题的文献调查（"调查 transformer 注意力变体"、"综述扩散模型的文献"）
- 多篇论文的综合（"关于 X 的最新论文说什么"、"比较 Y 论文的方法论"）
- 具有一致引用格式的系统综述（"用 APA 格式做 Z 的 SLR"）
- 关于某个主题的带注释书目索引
- 某个领域一段时间内的研究趋势概述

**不要**在以下情况下使用此技能：

- 用户只提供一篇论文并要求审查（使用 `academic-paper-review`）
- 用户提出的事实性问题不需要综合多个来源（直接回答）
- 用户想要没有学术严谨性的通用网络研究（使用标准网络搜索）

## 工作流程

工作流程有五个阶段。按顺序执行。

### 阶段 1：计划

在任何检索之前，先与用户确认以下内容。如果有任何不清楚的地方，问**一个**涵盖缺失部分的问题。不要一次问一个问题。

- **主题**：用简单英语描述的研究领域（例如，"transformer 注意力变体"）。
- **范围**：多少篇论文（默认 20 篇，上限 50 篇），可选时间窗口（例如"最近 2 年"），可选 arXiv 类别（例如 `cs.CL`、`cs.CV`）。
- **引用格式**：APA、IEEE 或 BibTeX（如果用户未指定且似乎不是为特定场所撰写，则默认为 APA）。
- **输出位置**：保存最终报告的位置（默认为 `/mnt/user-data/outputs/`）。

如果用户说"50+ 篇论文"，请礼貌地将其限制在 50 篇，并解释超过这个数量综合质量会迅速下降——对于更大的综述，他们应该按子主题分割。

### 阶段 2：搜索 arXiv

调用捆绑的搜索脚本。**不要**尝试通过其他方式抓取 arXiv，**不要**编写自己的 HTTP 客户端——此脚本正确处理 URL 编码、Atom XML 解析和 ID 规范化。

```bash
python /mnt/skills/public/systematic-literature-review/scripts/arxiv_search.py \
  "<topic>" \
  --max-results <N> \
  [--category <cat>] \
  [--sort-by relevance] \
  [--start-date YYYY-MM-DD] \
  [--end-date YYYY-MM-DD]
```

**重要——搜索前提取 2-3 个核心关键词。** 不要将用户的完整主题描述作为查询。在调用脚本之前，将主题 mentally reduce 为其最关键的 2-3 个术语。删除限定词如"in computer vision"、"for NLP"、"variants"、"recent"——那些属于 `--category` 或 `--start-date`，而不是查询字符串。

**查询措辞——保持简短。** 脚本将多词查询用双引号包裹以进行短语匹配。这意味着：

- `"diffusion models"` → 搜索精确短语 → 好，返回相关论文。
- `"diffusion models in computer vision"` → 搜索那个精确的 5 词短语 → **太具体，可能返回 0 结果**，因为很少有论文包含那个精确字符串。

使用 **2-3 个核心关键词** 作为查询，使用 `--category` 来缩小领域，而不是将领域名称塞入查询。示例：

| 用户说 | 好的查询 | 坏的查询 |
|---|---|---|
| "diffusion models in computer vision" | `"diffusion models" --category cs.CV` | `"diffusion models in computer vision"` |
| "transformer attention variants" | `"transformer attention"` | `"transformer attention variants in NLP"` |
| "graph neural networks for molecules" | `"graph neural networks" --category cs.LG` | `"graph neural networks for molecular property prediction"` |

脚本向 stdout 打印一个 JSON 数组。每篇论文有：`id`、`title`、`authors`、`abstract`、`published`、`updated`、`categories`、`pdf_url`、`abs_url`。

**排序策略**：

- **始终使用 `relevance` 排序**——arXiv 的 BM25 风格评分确保结果确实与用户的主题相关。`submittedDate` 排序返回该类别中最近提交的论文，无论主题相关性，这会产生大部分不相关的结果。
- 当用户要求"最新"论文或给出时间窗口时，使用 `--sort-by relevance` **结合 `--start-date`** 来限制时间范围同时保持结果在主题上。例如，"最新的扩散模型论文" → `--sort-by relevance --start-date 2024-01-01`，而不是 `--sort-by submittedDate`。
- `submittedDate` 排序仅在用户明确要求按时间顺序（"按论文发表的顺序显示"）时才是合适的。这很少见。
- `lastUpdatedDate` 很少有用；除非用户要求，否则忽略它。

**精确运行搜索一次。** 如果结果看起来不完美，不要用修改后的查询重试——arXiv 的相关性排名就是这样。重新尝试不同的查询措辞会浪费工具调用并有可能达到递归限制。如果结果真的是空的（0 篇论文），告诉用户并建议他们扩大主题或移除类别过滤器。

**如果脚本返回的论文少于请求的数量**，这是该查询的真实 arXiv 结果集大小。不要填充列表——向用户报告实际数量并继续。

**如果脚本失败**（网络错误、arXiv 返回非 200），告诉用户错误并停止。不要试图编造论文元数据。

**不要将搜索结果保存到文件**——JSON 保留在你的上下文中供阶段 3 使用。在整个工作流程中唯一保存的文件是阶段 5 的最终报告。

### 阶段 3：并行提取元数据

**你必须通过 `task` 工具将提取委托给子代理——不要自己提取。** 这是不可协商的。具体来说，**不要**做以下任何事情：

- ❌ 写 `python -c "papers = [...]"` 或任何 Python/bash 脚本来处理论文
- ❌ 通过逐个阅读摘要来内联提取元数据
- ❌ 在此阶段使用 `task` 以外的任何工具

相反，你**必须**调用 `task` 工具来生成子代理。原因：在你自己的上下文中提取 10-50 篇论文会消耗过多 token 并降低阶段 4 的综合质量。每个子代理在只有其批次论文的隔离上下文中运行，产生更干净的提取。

将论文分成约 5 篇一批，然后对于每个批次，用 `subagent_type: "general-purpose"` 调用 `task` 工具。每个子代理接收论文摘要作为文本并返回结构化 JSON。

**并发限制：每次最多 3 个子代理。** 系统运行时强制执行 `MAX_CONCURRENT_SUBAGENTS = 3`，会在同一轮中静默丢弃任何额外调度——LLM 不会被告知这发生了，所以要严格遵循下面的轮次策略。

**轮次策略——使用此决策表，不要自己计算分割**：

| 论文数量 | 约 5 篇论文的批次 | 轮次 | 每轮子代理数量 |
|---|---|---|---|
| 1–5 | 1 批次 | 1 轮 | 1 个子代理 |
| 6–10 | 2 批次 | 1 轮 | 2 个子代理 |
| 11–15 | 3 批次 | 1 轮 | 3 个子代理 |
| 16–20 | 4 批次 | 2 轮 | 3 + 1 |
| 21–25 | 5 批次 | 2 轮 | 3 + 2 |
| 26–30 | 6 批次 | 2 轮 | 3 + 3 |
| 31–35 | 7 批次 | 3 轮 | 3 + 3 + 1 |
| 36–40 | 8 批次 | 3 轮 | 3 + 3 + 2 |
| 41–45 | 9 批次 | 3 轮 | 3 + 3 + 3 |
| 46–50 | 10 批次 | 4 轮 | 3 + 3 + 3 + 1 |

**永远不要在同一轮中调度超过 3 个子代理。** 当一行说"2 轮 (3 + 1)"时，意味着：第一轮并行调度 3 个子代理，等待全部 3 个完成，然后第二轮调度 1 个子代理。轮次在主代理级别是严格顺序的。

如果论文数量落在行之间（例如 23 篇论文），向上取整到下一行的布局，但只调度你实际需要的批次数量——决策表给你的是形状，不是硬性规定。

**在主代理级别进行分批**：你已经在阶段 2 获得了每篇论文的摘要，所以每个子代理接收纯文本输入。子代理不需要访问网络或沙箱——它们唯一的工作是读取文本并返回 JSON。不要要求子代理重新运行 `arxiv_search.py`；那会浪费 token 并有可能触发速率限制。

**每个子代理接收的内容**，作为结构化提示：

```
执行此任务：从以下 arXiv 论文中提取结构化元数据和关键发现。

论文：
[论文 1]
arxiv_id: 1706.03762
title: Attention Is All You Need
authors: Ashish Vaswani, Noam Shazeer, ...
published: 2017-06-12
abstract: <完整摘要文本>

[论文 2]
arxiv_id: ...
...

对于每篇论文，返回一个包含这些字段的 JSON 对象：
- arxiv_id (字符串)
- title (字符串)
- authors (字符串列表)
- published_date (字符串, YYYY-MM-DD)
- research_question (1 句，论文解决的问题)
- methodology (1-2 句，他们如何解决)
- key_findings (3-5 个要点，他们实际发现的)
- limitations (1-2 句，他们承认的或明显缺失的)

将结果作为 JSON 数组返回，每篇论文一个对象，按输入顺序排列。不要在 JSON 之外包含任何文本——没有开场白，没有 markdown 代码块，只有数组。
```

**解析子代理结果**：任务工具返回带有固定前缀的字符串，如 `Task Succeeded. Result: [...JSON...]`。在尝试解析 JSON 之前，剥离 `Task Succeeded. Result: ` 前缀（或 `Task failed.` / `Task timed out.` 前缀）。如果一个批次失败或返回不可解析的 JSON，记录下来，注明受影响的论文，并继续处理剩余批次——不要因为一个坏批次而使整个综合失败。

所有轮次完成后，将每批数组展平为单个论文元数据对象列表，保留顺序。

### 阶段 4：综合和格式化

现在生成最终的 SLR 报告。这里发生两件事：跨论文综合（主题分析）和引用格式化。

**跨论文综合**：报告必须做得比列出论文更多。至少要确定：

- **主题**：整个集合中 3-6 个反复出现的研究方向、方法或问题框架。
- **趋同**：多篇论文同意的发现。
- **分歧**：论文得出不同结论或使用不兼容方法的地方。
- **差距**：文献集合尚未解决的问题（通常在"局限性"字段中明确说明）。

如果论文集太小或太异构以至于无法支持主题综合（例如，关于完全不同子主题的 5 篇论文），在报告中明确说明——不要强行构建不存在的主题。

**引用格式化**：确切的格式取决于用户偏好。只读取**与用户请求格式匹配的模板文件**，不是全部三个：

- [templates/apa.md](templates/apa.md)——APA 第 7 版。社会科学和大多数 CS 期刊的默认值。当用户请求 APA 或未指定格式时使用。
- [templates/ieee.md](templates/ieee.md)——IEEE 数字引用。当用户目标是 IEEE 会议或期刊，或明确要求 IEEE 时使用。
- [templates/bibtex.md](templates/bibtex.md)——BibTeX 条目。当用户提到 BibTeX、LaTeX 或想要机器可读引用时使用。**重要**：arXiv 论文引用为 `@misc`，不是 `@article`——BibTeX 模板明确涵盖了这一内容。

每个模板包含引用规则和完整报告结构（执行摘要、主题、每篇论文注释、参考文献、方法论部分）。严格按照模板的结构来写报告正文，然后用阶段 3 元数据填充内容。

### 阶段 5：保存和展示

将完整报告保存到 `/mnt/user-data/outputs/slr-<topic-slug>-<YYYYMMDD>.md`，其中 `<topic-slug>` 是主题的小写连字符版本（例如 `transformer-attention`）。然后调用 `present_files` 工具并提供该路径，以便用户可以下载。

**在聊天消息中**，显示一个简短预览，让用户立即看到价值，而无需打开文件：

1. **执行摘要**——报告顶部的 3-5 句话段落，原文照录。
2. **主题列表**——你在阶段 4 综合中确定的主题的项目符号列表（只有主题名称 + 一句话解释，不是完整的主题部分）。
3. **论文数量 + 文件指针**——例如，"包含 20 篇论文、每篇论文注释和格式化参考文献的完整报告已保存到 `slr-transformer-attention-20260409.md`。"

**不要**在内联中转储完整的 2000+ 字报告——每篇论文注释、参考文献和方法论部分属于文件内。预览是为了让用户一眼就能判断报告并决定是否打开它。

## 示例

**示例 1：典型 SLR 请求**

用户："对最近的 transformer 注意力变体进行系统文献综述，20 篇论文，APA 格式。"

你的流程：
1. 阶段 1：确认主题（transformer 注意力变体）、范围（20 篇论文，默认时间窗口）、格式（APA）。只在缺少内容时才问**一个**澄清问题（例如，"有什么特定的时间窗口吗，还是我默认最近 3 年？"）。
2. 阶段 2：`arxiv_search.py "transformer attention" --max-results 20 --sort-by relevance --start-date 2023-01-01`。
3. 阶段 3：20 篇论文 → 第 1 轮 = 3 个子代理 × 5 篇论文 = 15 篇覆盖，第 2 轮 = 1 个子代理 × 5 篇论文 = 5 篇覆盖。聚合。
4. 阶段 4：读取 `templates/apa.md`，使用其结构写报告，用阶段 3 元数据填充主题 + 每篇论文注释。
5. 阶段 5：保存到 `slr-transformer-attention-20260409.md`，调用 `present_files`。

**示例 2：小规模请求且有歧义**

用户："为我调查一些关于扩散模型的论文。"

你的流程：
1. 阶段 1："一些"是有歧义的。问一个问题："你想要多少篇论文——10、20 还是 30？还有引用格式偏好吗（默认是 APA）？"
2. 用户回复"10 篇，BibTeX"。
3. 阶段 2：`arxiv_search.py "diffusion models" --max-results 10 --category cs.CV`。
4. 阶段 3：10 篇论文 → 单轮，2 个子代理 × 5 篇论文。
5. 阶段 4：读取 `templates/bibtex.md`，用 `@misc` 条目格式化（不是 `@article`）。
6. 阶段 5：保存并展示。

**示例 3：超出范围请求**

用户："这里有一篇论文（https://arxiv.org/abs/1706.03762）。你能审查一下吗？"

这是单篇论文的同行评审，不是文献调查。不要使用此技能。改用 `academic-paper-review`。

## 注意事项

- **前提条件：`subagent_enabled` 必须为 `true`**。阶段 3 需要 `task` 工具进行并行元数据提取。此工具仅在运行时配置 `config.configurable.subagent_enabled` 中 `subagent_enabled` 设置为 `true` 时才会加载。没有它，`task` 工具不会出现在可用工具中，阶段 3 无法按设计执行。
- **默认只搜索 arXiv**。此技能不查询 Semantic Scholar、PubMed 或 Google Scholar。arXiv 涵盖了 CS/ML/物理/数学预印本的大部分内容，这是用户最常想要调查的。多源学术搜索属于专门的 MCP 服务器，而不是这个技能内部。
- **论文硬上限为 50 篇**。这与阶段 3 并发策略相关（每轮最多 3 个子代理，每批约 5 篇，最多约 3 轮）。大于 50 篇论文的综述综合质量会下降，最好按子主题分割。
- **阶段 3 需要启用子代理**。此技能的并行提取步骤硬性要求 `task` 工具，该工具仅在运行时 `subagent_enabled=true` 时可用。如果子代理不可用，不要声称执行阶段 3 并行计划；而是告诉用户必须启用子代理才能完成完整工作流程，或者提出将请求缩小/分割成更小规模的手动审查。
- **子代理结果是字符串，不是对象**。在解析 JSON 载荷之前，始终剥离 `Task Succeeded. Result: ` / `Task failed.` / `Task timed out.` 前缀。
- **`id` 字段是裸 arXiv id**（例如 `1706.03762`），不是 URL 也不是带版本后缀的。如果需要，`abs_url` / `pdf_url` 包含完整的 URL。
- **综合，而非列表**。最终报告必须识别主题并比较论文间的发现。只逐个列出论文的报告是失败模式——如果你找不到主题，明确说出来，而不是伪造主题。
